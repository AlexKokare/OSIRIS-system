## 1. Каноническое ядро архитектуры

### 1.1. Базовое решение: одна сфера = один 512-битный регистр

На уровне `TLU` одна рабочая сфера — это **один ZMM-регистр AVX-512**, разбитый на 8 дорожек по 64 бита.

### 1.2. Каноническая раскладка дорожек

| Lane | Смысл | Роль |
|---|---|---|
| 0 | `+X` | экспансия / рост |
| 1 | `+Y` | синтез / сцепка |
| 2 | `+Z` | память / резонанс |
| 3 | `-X` | сжатие / дебет |
| 4 | `-Y` | разрез / деление |
| 5 | `-Z` | зеркало / инверсия |
| 6 | `PHASE` | фазовый корень / такт 1:3 |
| 7 | `ECHO` | эхо-подпись / zero-lock / контроль |

Это и есть окончательный ответ на вопрос «4 или 6 осей?»:

- **4 оси** остаются как подмодель для вращений и вложенных переходов.
- **6 осей** становятся рабочим топологическим каркасом ядра.

Иначе говоря:

- 4D — это **внутренняя геометрия перехода**,
- 6 осей — это **операционная геометрия исполнения**.

### 1.3. Одна сфера = 3 прямые + 3 зеркальные плоскости

Одна сфера задаётся как:

- три прямые плоскости: `+X`, `+Y`, `+Z`
- три зеркальные плоскости: `-X`, `-Y`, `-Z`

Поэтому **одна сфера уже содержит полный цикл “прямая реальность + зеркальная реальность”**.

### 1.4. Три рабочие сферы системы

На уровне исполнения нужны три базовые сферы:

- `ZMM0` — **Контекст C**
- `ZMM1` — **Состояние A**
- `ZMM2` — **Состояние B / транзакция / возмущение**

Это снимает противоречие между «одной сферой» и «тремя сферами»:

- **одна сфера** — минимальный объект,
- **три сферы** — минимальная сцена исполнения.

## 4. Фазовая единица 1:3

### 4.1. Что именно фиксируем

Отношение `1:3` больше не считается дробью. Оно фиксируется как **структурный трехтактный проход**.

Базовая триада прямых осей:

- `+X -> +Y -> +Z -> +X`

Зеркальная триада:

- `-X -> -Y -> -Z -> -X`

### 4.2. Что делает фазовый движок

Фазовый движок не вычисляет десятичное `1/3`.
Он делает:

1. сдвиг внутри триады,
2. второй сдвиг,
3. третий сдвиг,
4. проверку возврата в топологически эквивалентное состояние.

### 4.3. Практическая формула фазы

Для ядра это значит:

- одна фаза = один допустимый сдвиг внутри триады,
- три фазы = полный цикл,
- несовпадение после трёх фаз = внешнее искажение.

## 5. Облако событий: каноническая модель Блока III

Это центральная часть доработки файла 33.

### 5.1. Что такое Облако событий

`Облако событий` — это не абстракция, а **конкретная матрица 6x6**, где каждая ось `A`
сталкивается с каждой осью `B`.

Итого:

- 6 осей сферы A
- 6 осей сферы B
- 36 микро-состояний

### 5.2. Как это кладётся в регистры

Для облака используются `ZMM3-ZMM8`.

Каждый регистр хранит **одну ось A против всех 6 осей B**.

| Регистр | Что хранит |
|---|---|
| `ZMM3` | `A(+X)` против `B(+X,+Y,+Z,-X,-Y,-Z)` |
| `ZMM4` | `A(+Y)` против `B(...)` |
| `ZMM5` | `A(+Z)` против `B(...)` |
| `ZMM6` | `A(-X)` против `B(...)` |
| `ZMM7` | `A(-Y)` против `B(...)` |
| `ZMM8` | `A(-Z)` против `B(...)` |

Внутри каждого регистра:

- lanes `0..5` — 6 микро-состояний
- lane `6` — локальная фазовая подпись
- lane `7` — локальный контроль / эхо

### 5.3. Что это даёт

Это даёт абсолютно чёткую инженерную картину:

- 6 регистров × 6 полезных дорожек = 36 микро-состояний,
- облако полностью влезает в выделенный кластер регистров,
- каждая перекомбинация локализована,
- схлопывание облака можно делать ступенчато.

### 5.4. Жизненный цикл Облака событий

#### Шаг 0. Нормализация

- `ZMM1` и `ZMM2` приводятся к канонической раскладке осей.

#### Шаг 1. Фазовое расщепление

- триады `(+X,+Y,+Z)` и `(-X,-Y,-Z)` прогоняются через фазовый цикл 1:3.

#### Шаг 2. Вброс в облако

- одна ось A тиражируется по рабочему регистру,
- оси B переставляются пермутацией,
- получается семейство из 6 столкновений.

#### Шаг 3. Локальный резонанс

Для каждого микро-состояния считается:

- совместимость,
- зеркальность,
- память/эхо,
- право на выживание.

#### Шаг 4. Зеркальная отсечка

Всё, что не имеет пути к `Zero-Lock`, инвертируется через `-Z` и удаляется из облака.

#### Шаг 5. Схлопывание

Из 36 микро-состояний должен остаться **один допустимый результирующий вектор**
или нулевой отказ.

---

## 6. Деление через пирамиды, вращение через кубы

Это решение примиряет две сильные идеи из материалов.

### 6.1. Когда использовать пирамиды

**Пирамидальный режим** применяется, когда нужно:

- вывернуть состояние по конкретному вектору,
- сделать направленную инверсию,
- открыть путь внутрь сферы,
- провести “операцию разреза”.

Пирамида — это **оператор выворачивания**.

### 6.2. Когда использовать кубы

**Кубический режим** применяется, когда нужно:

- повернуть структуру,
- перейти между плоскостями,
- поменять систему наблюдения,
- сделать 4D-переориентацию без разрушения содержимого.

Куб — это **оператор переориентации**.

### 6.3. Формула совместного использования

- `PYRAMID = направленное выворачивание`
- `CUBE = топологический поворот`

То есть:

- деление / инверсия / вскрытие — через пирамиды,
- вращение / перестановка / смена вида — через кубы.

# OSIRIS-ASM — доработанная версия файла 33

## 0. Зачем этот файл

Этот документ фиксирует **каноническую архитектуру** для дальнейшей сборки `OSIRIS-ASM`.
Его задача — убрать разночтения между идеями про 4/6 осей, 8-битные сферы, 3x3/4x4 плоскости,
кубы, пирамиды, циклы, память и AVX-512, и свести их в **одну инженерную схему**, пригодную для
дорожной карты на 1000+ строк ассемблера.

---

## 1. Каноническое ядро архитектуры

### 1.1. Базовое решение: одна сфера = один 512-битный регистр

На уровне `TLU` одна рабочая сфера — это **один ZMM-регистр AVX-512**, разбитый на 8 дорожек по 64 бита.

### 1.2. Каноническая раскладка дорожек

| Lane | Смысл | Роль |
|---|---|---|
| 0 | `+X` | экспансия / рост |
| 1 | `+Y` | синтез / сцепка |
| 2 | `+Z` | память / резонанс |
| 3 | `-X` | сжатие / дебет |
| 4 | `-Y` | разрез / деление |
| 5 | `-Z` | зеркало / инверсия |
| 6 | `PHASE` | фазовый корень / такт 1:3 |
| 7 | `ECHO` | эхо-подпись / zero-lock / контроль |

Это и есть окончательный ответ на вопрос «4 или 6 осей?»:

- **4 оси** остаются как подмодель для вращений и вложенных переходов.
- **6 осей** становятся рабочим топологическим каркасом ядра.

Иначе говоря:

- 4D — это **внутренняя геометрия перехода**,
- 6 осей — это **операционная геометрия исполнения**.

### 1.3. Одна сфера = 3 прямые + 3 зеркальные плоскости

Одна сфера задаётся как:

- три прямые плоскости: `+X`, `+Y`, `+Z`
- три зеркальные плоскости: `-X`, `-Y`, `-Z`

Поэтому **одна сфера уже содержит полный цикл “прямая реальность + зеркальная реальность”**.

### 1.4. Три рабочие сферы системы

На уровне исполнения нужны три базовые сферы:

- `ZMM0` — **Контекст C**
- `ZMM1` — **Состояние A**
- `ZMM2` — **Состояние B / транзакция / возмущение**

Это снимает противоречие между «одной сферой» и «тремя сферами»:

- **одна сфера** — минимальный объект,
- **три сферы** — минимальная сцена исполнения.

---

## 2. Четыре уровня представления

Чтобы не смешивать масштабы, архитектура делится на 4 слоя.

### Уровень A — микрослой (8 бит)

Это локальный логический узел:

- 4 бита состояния
- 4 бита модификации

Этот слой нужен для:

- локальных триггеров
- префиксной логики
- резонансных мини-циклов
- описания зерна памяти

### Уровень B — плоскость 3x3 и 4x4

Два типа плоскостей сохраняются, но получают разные роли:

#### Плоскость 3x3

Это **логическое ядро**:

- 8 направлений
- 1 центр-якорь

Используется для:

- поворотов
- выбора локального направления
- якоря симметрии

#### Плоскость 4x4

Это **адресная и визуализирующая шина**:

- 16 ячеек
- удобно ложится на 4-битную индексацию
- хорошо упаковывается в линейную память

Используется для:

- адресации
- периферийной маски
- раскладки сектора сферы

### Уровень C — вычислительная сфера (512 бит)

Это основной объект `OSIRIS-ASM`, на котором работает TLU.

### Уровень D — облако событий

Это **результат столкновения двух сфер A и B по всем 6 осям**.

Именно здесь рождаются:

- 36 микро-состояний
- фильтрация мусора
- zero-lock
- окончательный вектор коммита

## 3. Главное разрешение противоречий

### 3.1. Что считать каноном

Каноническая схема такая:

1. **8-битный узел** — минимальная клетка логики.
2. **3x3 / 4x4** — локальная геометрия и адресная маска.
3. **512-битная сфера** — главный объект TLU.
4. **Три сферы** — сцена вычисления.
5. **36 состояний** — облако событий.

### 3.2. Что считать вторичным

Вторичными считаются:

- буквальные трактовки всех промежуточных геометрий как отдельных ISA-объектов,
- смешение слоя визуализации с ядром исполнения,
- попытка заставить каждую идею сразу стать железным примитивом.

### 3.3. Практический вывод

В коде ядра первым классом сущностей становятся:

- `SPHERE`
- `PHASE`
- `MIRROR`
- `CLOUD`
- `COMMIT`

А плоскости 3x3/4x4 и 8-битные схемы остаются **подмоделями кодирования**.

---

## 4. Фазовая единица 1:3

### 4.1. Что именно фиксируем

Отношение `1:3` больше не считается дробью. Оно фиксируется как **структурный трехтактный проход**.

Базовая триада прямых осей:

- `+X -> +Y -> +Z -> +X`

Зеркальная триада:

- `-X -> -Y -> -Z -> -X`

### 4.2. Что делает фазовый движок

Фазовый движок не вычисляет десятичное `1/3`.
Он делает:

1. сдвиг внутри триады,
2. второй сдвиг,
3. третий сдвиг,
4. проверку возврата в топологически эквивалентное состояние.

### 4.3. Практическая формула фазы

Для ядра это значит:

- одна фаза = один допустимый сдвиг внутри триады,
- три фазы = полный цикл,
- несовпадение после трёх фаз = внешнее искажение.

---

## 5. Облако событий: каноническая модель Блока III

Это центральная часть доработки файла 33.

### 5.1. Что такое Облако событий

`Облако событий` — это не абстракция, а **конкретная матрица 6x6**, где каждая ось `A`
сталкивается с каждой осью `B`.

Итого:

- 6 осей сферы A
- 6 осей сферы B
- 36 микро-состояний

### 5.2. Как это кладётся в регистры

Для облака используются `ZMM3-ZMM8`.

Каждый регистр хранит **одну ось A против всех 6 осей B**.

| Регистр | Что хранит |
|---|---|
| `ZMM3` | `A(+X)` против `B(+X,+Y,+Z,-X,-Y,-Z)` |
| `ZMM4` | `A(+Y)` против `B(...)` |
| `ZMM5` | `A(+Z)` против `B(...)` |
| `ZMM6` | `A(-X)` против `B(...)` |
| `ZMM7` | `A(-Y)` против `B(...)` |
| `ZMM8` | `A(-Z)` против `B(...)` |

Внутри каждого регистра:

- lanes `0..5` — 6 микро-состояний
- lane `6` — локальная фазовая подпись
- lane `7` — локальный контроль / эхо

### 5.3. Что это даёт

Это даёт абсолютно чёткую инженерную картину:

- 6 регистров × 6 полезных дорожек = 36 микро-состояний,
- облако полностью влезает в выделенный кластер регистров,
- каждая перекомбинация локализована,
- схлопывание облака можно делать ступенчато.

### 5.4. Жизненный цикл Облака событий

#### Шаг 0. Нормализация

- `ZMM1` и `ZMM2` приводятся к канонической раскладке осей.

#### Шаг 1. Фазовое расщепление

- триады `(+X,+Y,+Z)` и `(-X,-Y,-Z)` прогоняются через фазовый цикл 1:3.

#### Шаг 2. Вброс в облако

- одна ось A тиражируется по рабочему регистру,
- оси B переставляются пермутацией,
- получается семейство из 6 столкновений.

#### Шаг 3. Локальный резонанс


; =============================================================================
; ПРОЕКТ: OSIRIS-ASM
; СТАТУС: КАРКАС СТУПЕНИ 0/1
; НАЗНАЧЕНИЕ: синхронный asm-скелет к доработанному файлу 33
; АРХИТЕКТУРА: x86-64 / AVX-512

global _start_tlu

%define L_NY    4

section .data
    align 64
    ABS_ZERO            times 64 db 0x00

    align 64
    MIRROR_MASK         times 64 db 0xFF

    align 64
    ; lane-map: +X,+Y,+Z,-X,-Y,-Z,PHASE,ECHO
    SPHERE_LAYOUT       dq 0, 1, 2, 3, 4, 5, 6, 7

    align 64
    ; циклический поворот прямой триады (+X,+Y,+Z)
    PHASE_DIR_ROT_1     dq 1, 2, 0, 3, 4, 5, 6, 7
    PHASE_DIR_ROT_2     dq 2, 0, 1, 3, 4, 5, 6, 7

    align 64
    ; циклический поворот зеркальной триады (-X,-Y,-Z)

    PHASE_MIR_ROT_2     dq 0, 1, 2, 5, 3, 4, 6, 7

    align 64
    ; рабочие индексы для Блока III: B раскладывается по всем 6 осям
    B_ALL_AXES          dq L_PX, L_PY, L_PZ, L_NX, L_NY, L_NZ, L_PHASE, L_ECHO

section .text

; -----------------------------------------------------------------------------
; БЛОК I: ИНИЦИАЛИЗАЦИЯ ПРОСТРАНСТВА
; ZMM0 = CONTEXT C
; ZMM1 = SPHERE A
; ZMM2 = SPHERE B
; ZMM3-ZMM8 = ОБЛАКО СОБЫТИЙ (6 x 6)
; -----------------------------------------------------------------------------
_start_tlu:
    cli

    ; очистка базовых сфер и рабочего облака
    vpxorq zmm0, zmm0, zmm0
    vpxorq zmm1, zmm1, zmm1
    vpxorq zmm2, zmm2, zmm2
    vpxorq zmm3, zmm3, zmm3
    vpxorq zmm4, zmm4, zmm4
    vpxorq zmm5, zmm5, zmm5
    vpxorq zmm6, zmm6, zmm6
    vpxorq zmm7, zmm7, zmm7
    vpxorq zmm8, zmm8, zmm8

    ; загрузка эталонного контекста
    lea rax, [rel ABS_ZERO]
    vmovdqa64 zmm0, [rax]

    ; далее должны вызываться этапы:
    ; 1. tlu_phase_engine
    ; 2. tlu_expand_cloud
    ; 3. tlu_filter_cloud
    ; 4. tlu_zero_lock
    ; 5. tlu_commit

    ret

; -----------------------------------------------------------------------------
; БЛОК II: ДВИЖОК ФАЗЫ 1:3
; канон: (+X,+Y,+Z) и (-X,-Y,-Z) крутятся как две отдельные триады
; -----------------------------------------------------------------------------
tlu_phase_engine:

    ; - сверить результат с lane ECHO

; -----------------------------------------------------------------------------
; БЛОК III: ОБЛАКО СОБЫТИЙ
; ZMM3 = A(+X) против всех 6 осей B
; ZMM4 = A(+Y) против всех 6 осей B
; ZMM5 = A(+Z) против всех 6 осей B
; ZMM6 = A(-X) против всех 6 осей B
; ZMM7 = A(-Y) против всех 6 осей B
; ZMM8 = A(-Z) против всех 6 осей B
; lanes 0..5 = 36 микро-состояний
; lane 6 = локальная фаза
; lane 7 = локальное эхо
; -----------------------------------------------------------------------------
tlu_expand_cloud:

    ret

tlu_filter_cloud:

; -----------------------------------------------------------------------------
; БЛОК IV: СФЕРИЧЕСКАЯ АРИФМЕТИКА

tlu_invert_sphere:

; -----------------------------------------------------------------------------
; БЛОК V: ZERO-LOCK И COMMIT

tlu_zero_lock:

; =============================================================================
; ПРОЕКТ: OSIRIS-ASM
; СТАДИЯ: СТУПЕНЬ 0/1 — КАНОНИЧЕСКИЙ КАРКАС TLU
; ЦЕЛЬ:
;   1. Зафиксировать 6-осевую сферу: +X,+Y,+Z,-X,-Y,-Z,PHASE,ECHO
;   2. Реализовать фазовый цикл 1:3 как структурный трехтактный проход
;   3. Развернуть Облако Событий 6x6 (36 микро-состояний)
;   4. Ввести fail-closed шлюз: инъекция должна самозамкнуться,
;      иначе она изымается и вектор дальше не проходит
;   5. Запретить фиксацию результирующего состояния в радиусе 0
; =============================================================================

bits 64
default rel

global _start_tlu

%define L_PX            0
%define L_PY            1
%define L_PZ            2
%define L_NX            3
%define L_NY            4
%define L_NZ            5
%define L_PHASE         6
%define L_ECHO          7

%define SPHERE_BYTES    64
%define LANE_BYTES      8

%define PHASE_OK        0x0000000000000003
%define PHASE_REJECT    0x00000000000000FF
%define ECHO_OK         0xE011C105E011C105
%define ECHO_REJECT     0xDEAD0000DEAD0000

%macro TLU_CLEAR 1
    vpxorq %1, %1, %1
%endmacro

section .data align=64
    align 64
    ABS_ZERO:
        times SPHERE_BYTES db 0x00

    align 64
    MIRROR_MASK:
        times SPHERE_BYTES db 0xFF

    align 64
    CLOUD_PAYLOAD_MASK:
        dq -1, -1, -1, -1, -1, -1, 0, 0

    align 64
    DIRECT_TRIAD_MASK:
        dq -1, -1, -1, 0, 0, 0, 0, 0

    align 64
    MIRROR_TRIAD_MASK:
        dq 0, 0, 0, -1, -1, -1, 0, 0

    align 64
    PHASE_ROOT:
        dq 0, 0, 0, 0, 0, 0, PHASE_OK, 0

    align 64
    ECHO_ROOT:
        dq 0, 0, 0, 0, 0, 0, 0, ECHO_OK

    align 64
    SPHERE_LAYOUT:
        dq L_PX, L_PY, L_PZ, L_NX, L_NY, L_NZ, L_PHASE, L_ECHO

    ; Будущие таблицы для аппаратных VPERMQ/VPERMT2Q-перекомбинаций.
    ; Пока каркас работает через явное облако и скалярную перестановку буферов.
    align 64
    B_FAMILY_0:
        dq L_PX, L_PY, L_PZ, L_NX, L_NY, L_NZ, L_PHASE, L_ECHO
    B_FAMILY_1:
        dq L_PY, L_PZ, L_PX, L_NY, L_NZ, L_NX, L_PHASE, L_ECHO
    B_FAMILY_2:
        dq L_PZ, L_PX, L_PY, L_NZ, L_NX, L_NY, L_PHASE, L_ECHO
    B_FAMILY_3:
        dq L_NX, L_NY, L_NZ, L_PX, L_PY, L_PZ, L_PHASE, L_ECHO
    B_FAMILY_4:
        dq L_NY, L_NZ, L_NX, L_PY, L_PZ, L_PX, L_PHASE, L_ECHO
    B_FAMILY_5:
        dq L_NZ, L_NX, L_NY, L_PZ, L_PX, L_PY, L_PHASE, L_ECHO

section .bss align=64
    align 64
    SPHERE_A_SCRATCH:   resb SPHERE_BYTES
    SPHERE_B_SCRATCH:   resb SPHERE_BYTES
    PHASE_A_SCRATCH:    resb SPHERE_BYTES
    PHASE_B_SCRATCH:    resb SPHERE_BYTES
    PHASE_A_ORIGINAL:   resb SPHERE_BYTES
    PHASE_B_ORIGINAL:   resb SPHERE_BYTES
    CLOUD_ROW_0:        resb SPHERE_BYTES
    CLOUD_ROW_1:        resb SPHERE_BYTES
    CLOUD_ROW_2:        resb SPHERE_BYTES
    CLOUD_ROW_3:        resb SPHERE_BYTES
    CLOUD_ROW_4:        resb SPHERE_BYTES
    CLOUD_ROW_5:        resb SPHERE_BYTES
    CLOUD_UNION:        resb SPHERE_BYTES
    CLOUD_CORE:         resb SPHERE_BYTES
    CLOUD_RESIDUE:      resb SPHERE_BYTES
    RESULT_SPHERE:      resb SPHERE_BYTES
    QUARANTINE_SPHERE:  resb SPHERE_BYTES
    DIVIDE_SCRATCH:     resb SPHERE_BYTES

section .text

; -----------------------------------------------------------------------------
; БЛОК I. ИНИЦИАЛИЗАЦИЯ ПРОСТРАНСТВА
; ВХОД:
;   ZMM1 = Сфера A
;   ZMM2 = Сфера B / кандидат на транзакцию / инъекцию
; ВЫХОД:
;   ZMM1 = закоммиченный результат или ноль
; -----------------------------------------------------------------------------
_start_tlu:
%ifdef OSIRIS_RING0
    cli
%endif

    ; ZMM0 = Абсолютный Ноль, остальные рабочие регистры очищаем.
    vmovdqa64 zmm0, [ABS_ZERO]
    TLU_CLEAR zmm3
    TLU_CLEAR zmm4
    TLU_CLEAR zmm5
    TLU_CLEAR zmm6
    TLU_CLEAR zmm7
    TLU_CLEAR zmm8
    TLU_CLEAR zmm9
    TLU_CLEAR zmm10
    TLU_CLEAR zmm11
    TLU_CLEAR zmm12
    TLU_CLEAR zmm13
    TLU_CLEAR zmm14
    TLU_CLEAR zmm15
    vmovdqa64 zmm16, [CLOUD_PAYLOAD_MASK]
    vmovdqa64 zmm17, [MIRROR_MASK]
    vmovdqa64 zmm18, [PHASE_ROOT]
    vmovdqa64 zmm19, [ECHO_ROOT]

    call tlu_normalize_spheres
    call tlu_phase_engine
    jc .reject

    call tlu_expand_cloud
    call tlu_self_closure_gate
    jc .reject

    call tlu_filter_cloud
    call tlu_zero_lock
    jc .reject

    call tlu_commit
    jmp .done

.reject:
    call tlu_quarantine

.done:
%ifdef OSIRIS_RING0
    sti
%endif
    ret

; -----------------------------------------------------------------------------
; БЛОК II. НОРМАЛИЗАЦИЯ СФЕР
; Канонизируем фазу/эхо. История осей не стирается.
; -----------------------------------------------------------------------------
tlu_normalize_spheres:
    vmovdqa64 [SPHERE_A_SCRATCH], zmm1
    vmovdqa64 [SPHERE_B_SCRATCH], zmm2

    mov qword [SPHERE_A_SCRATCH + (L_PHASE * LANE_BYTES)], PHASE_OK
    mov qword [SPHERE_B_SCRATCH + (L_PHASE * LANE_BYTES)], PHASE_OK
    mov qword [SPHERE_A_SCRATCH + (L_ECHO  * LANE_BYTES)], 0
    mov qword [SPHERE_B_SCRATCH + (L_ECHO  * LANE_BYTES)], 0

    vmovdqa64 zmm1, [SPHERE_A_SCRATCH]
    vmovdqa64 zmm2, [SPHERE_B_SCRATCH]
    ret

; -----------------------------------------------------------------------------
; БЛОК III. ДВИЖОК ФАЗЫ 1:3
; Инвариант: после трех проходов триады должны вернуться в исходное состояние.
; Иначе состояние считается внешне искаженным.
; -----------------------------------------------------------------------------
tlu_phase_engine:
    vmovdqa64 [PHASE_A_SCRATCH], zmm1
    vmovdqa64 [PHASE_A_ORIGINAL], zmm1
    vmovdqa64 [PHASE_B_SCRATCH], zmm2
    vmovdqa64 [PHASE_B_ORIGINAL], zmm2

    ; Проверка A
    lea rdi, [PHASE_A_SCRATCH]
    call tlu_rotate_direct_buffer
    call tlu_rotate_mirror_buffer
    call tlu_rotate_direct_buffer
    call tlu_rotate_mirror_buffer
    call tlu_rotate_direct_buffer
    call tlu_rotate_mirror_buffer
    lea rdi, [PHASE_A_SCRATCH]
    lea rsi, [PHASE_A_ORIGINAL]
    call tlu_compare_64
    jc .phase_fail

    ; Проверка B — именно здесь кандидат обязан доказать самозамыкание.
    lea rdi, [PHASE_B_SCRATCH]
    call tlu_rotate_direct_buffer
    call tlu_rotate_mirror_buffer
    call tlu_rotate_direct_buffer
    call tlu_rotate_mirror_buffer
    call tlu_rotate_direct_buffer
    call tlu_rotate_mirror_buffer
    lea rdi, [PHASE_B_SCRATCH]
    lea rsi, [PHASE_B_ORIGINAL]
    call tlu_compare_64
    jc .phase_fail

    ; Возвращаем исходные рабочие состояния в регистры.
    vmovdqa64 zmm1, [PHASE_A_ORIGINAL]
    vmovdqa64 zmm2, [PHASE_B_ORIGINAL]
    clc
    ret

.phase_fail:
    stc
    ret

; -----------------------------------------------------------------------------
; БЛОК IV. ОБЛАКО СОБЫТИЙ 6x6
; Каждая строка: одна ось A против всех шести осей B.
; lanes 0..5  = 36 микро-состояний
; lane  6     = фаза (позже проставляется zero-lock)
; lane  7     = эхо  (позже проставляется zero-lock)
; -----------------------------------------------------------------------------
tlu_expand_cloud:
    vmovdqa64 [SPHERE_A_SCRATCH], zmm1
    vmovdqa64 [SPHERE_B_SCRATCH], zmm2

    vpbroadcastq zmm3, qword [SPHERE_A_SCRATCH + (L_PX * LANE_BYTES)]
    vpxorq zmm3, zmm3, zmm2
    vpandq zmm3, zmm3, zmm16
    vmovdqa64 [CLOUD_ROW_0], zmm3

    vpbroadcastq zmm4, qword [SPHERE_A_SCRATCH + (L_PY * LANE_BYTES)]
    vpxorq zmm4, zmm4, zmm2
    vpandq zmm4, zmm4, zmm16
    vmovdqa64 [CLOUD_ROW_1], zmm4

    vpbroadcastq zmm5, qword [SPHERE_A_SCRATCH + (L_PZ * LANE_BYTES)]
    vpxorq zmm5, zmm5, zmm2
    vpandq zmm5, zmm5, zmm16
    vmovdqa64 [CLOUD_ROW_2], zmm5

    vpbroadcastq zmm6, qword [SPHERE_A_SCRATCH + (L_NX * LANE_BYTES)]
    vpxorq zmm6, zmm6, zmm2
    vpandq zmm6, zmm6, zmm16
    vmovdqa64 [CLOUD_ROW_3], zmm6

    vpbroadcastq zmm7, qword [SPHERE_A_SCRATCH + (L_NY * LANE_BYTES)]
    vpxorq zmm7, zmm7, zmm2
    vpandq zmm7, zmm7, zmm16
    vmovdqa64 [CLOUD_ROW_4], zmm7

    vpbroadcastq zmm8, qword [SPHERE_A_SCRATCH + (L_NZ * LANE_BYTES)]
    vpxorq zmm8, zmm8, zmm2
    vpandq zmm8, zmm8, zmm16
    vmovdqa64 [CLOUD_ROW_5], zmm8
    ret

; -----------------------------------------------------------------------------
; БЛОК V. ШЛЮЗ САМОЗАМЫКАНИЯ
; До следующего шага допускается только тот кандидат, который уже доказал
; возврат в собственную фазу. Здесь повторно блокируем незамкнутый радиус 0.
; -----------------------------------------------------------------------------
tlu_self_closure_gate:
    ; Кандидат не имеет права быть тождественен абсолютному нулю,
    ; иначе он может зафиксировать систему в зависшем центре.
    vmovdqa64 [SPHERE_B_SCRATCH], zmm2
    lea rdi, [SPHERE_B_SCRATCH]
    call tlu_or_payload_6qwords
    test rax, rax
    jz .gate_fail

    ; Фазовое доказательство уже пройдено в tlu_phase_engine.
    ; Повторно разрешаем движение только ненулевому, самозамкнутому кандидату.
    clc
    ret

.gate_fail:
    stc
    ret

; -----------------------------------------------------------------------------
; БЛОК VI. ФИЛЬТРАЦИЯ ОБЛАКА
; Берем только устойчивое пересечение прямой и зеркальной ветви.
; Остальное считаем остатком/инъекцией и складываем в residue.
; -----------------------------------------------------------------------------
tlu_filter_cloud:
    ; UNION = всё, что вообще проявилось в облаке.
    vmovdqa64 zmm12, zmm3
    vporq zmm12, zmm12, zmm4
    vporq zmm12, zmm12, zmm5
    vporq zmm12, zmm12, zmm6
    vporq zmm12, zmm12, zmm7
    vporq zmm12, zmm12, zmm8
    vpandq zmm12, zmm12, zmm16
    vmovdqa64 [CLOUD_UNION], zmm12

    ; CORE = только то, что выжило одновременно в прямой и зеркальной геометрии.
    vmovdqa64 zmm13, zmm3
    vpandq zmm13, zmm13, zmm6
    vmovdqa64 zmm14, zmm4
    vpandq zmm14, zmm14, zmm7
    vmovdqa64 zmm15, zmm5
    vpandq zmm15, zmm15, zmm8
    vporq zmm13, zmm13, zmm14
    vporq zmm13, zmm13, zmm15
    vpandq zmm13, zmm13, zmm16
    vmovdqa64 [CLOUD_CORE], zmm13

    ; RESIDUE = UNION - CORE. Пока residue не очищен, состояние не пойдет дальше.
    vmovdqa64 zmm14, zmm13
    vpxorq zmm14, zmm14, zmm17
    vpandq zmm12, zmm12, zmm14
    vpandq zmm12, zmm12, zmm16
    vmovdqa64 [CLOUD_RESIDUE], zmm12
    ret

; -----------------------------------------------------------------------------
; БЛОК VII. ZERO-LOCK
; Если residue не обнулен — отклоняем.
; Если core пустой — отклоняем (запрет зависания в радиусе 0).
; Иначе проставляем phase/echo и готовим коммит.
; -----------------------------------------------------------------------------
tlu_zero_lock:
    lea rdi, [CLOUD_RESIDUE]
    call tlu_or_payload_6qwords
    test rax, rax
    jnz .zero_fail

    lea rdi, [CLOUD_CORE]
    call tlu_or_payload_6qwords
    test rax, rax
    jz .zero_fail

    vmovdqa64 zmm13, [CLOUD_CORE]
    vporq zmm13, zmm13, zmm18
    vporq zmm13, zmm13, zmm19
    vmovdqa64 [RESULT_SPHERE], zmm13
    clc
    ret

.zero_fail:
    stc
    ret

; -----------------------------------------------------------------------------
; БЛОК VIII. COMMIT / QUARANTINE
; -----------------------------------------------------------------------------
tlu_commit:
    vmovdqa64 zmm1, [RESULT_SPHERE]
    vmovdqa64 [SPHERE_A_SCRATCH], zmm1
    ret

tlu_quarantine:
    vmovdqa64 [QUARANTINE_SPHERE], zmm2
    mov qword [QUARANTINE_SPHERE + (L_PHASE * LANE_BYTES)], PHASE_REJECT
    mov qword [QUARANTINE_SPHERE + (L_ECHO  * LANE_BYTES)], ECHO_REJECT
    TLU_CLEAR zmm1
    TLU_CLEAR zmm2
    TLU_CLEAR zmm3
    TLU_CLEAR zmm4
    TLU_CLEAR zmm5
    TLU_CLEAR zmm6
    TLU_CLEAR zmm7
    TLU_CLEAR zmm8
    ret

; -----------------------------------------------------------------------------
; БЛОК IX. СФЕРИЧЕСКАЯ АРИФМЕТИКА — БАЗОВЫЕ ОПЕРАТОРЫ
; Это рабочие заготовки следующего шага, чтобы дорожная карта уже была в коде.
; -----------------------------------------------------------------------------
tlu_invert_sphere:
    ; Вход:  zmm2 = Сфера
    ; Выход: zmm14 = Инвертированная сфера
    vpxorq zmm14, zmm2, zmm17
    ret

tlu_divide_sphere:
    ; Структурное деление: зеркало + маска + popcount.
    ; Вход:  zmm1 = dividend, zmm2 = divisor
    ; Выход: rax  = грубая мера совпадения/отношения
    ;         zmm15 = маска совпадения
    call tlu_invert_sphere
    vpandq zmm15, zmm1, zmm14
    vmovdqa64 [DIVIDE_SCRATCH], zmm15

    xor rax, rax
    mov rcx, [DIVIDE_SCRATCH + 0]
    popcnt rdx, rcx
    add rax, rdx
    mov rcx, [DIVIDE_SCRATCH + 8]
    popcnt rdx, rcx
    add rax, rdx

# OSIRIS-ASM — каноническая дорожная карта ядра

## 1. Что зафиксировано по материалам `32.txt` и `33 выжимка.txt`

Ниже — не абстрактная теория, а **рабочий канон**, от которого уже можно строить 1000+ строк NASM/x86-64 кода.

### 1.1. Неподвижные архитектурные решения

1. **Главный объект исполнения** — одна 512-битная сфера в одном `ZMM` регистре.
2. **Канонический lane-map**:
   - `0 = +X`
   - `1 = +Y`
   - `2 = +Z`
   - `3 = -X`
   - `4 = -Y`
   - `5 = -Z`
   - `6 = PHASE`
   - `7 = ECHO`
3. **Минимальная сцена исполнения**:
   - `ZMM0 = CONTEXT / C`
   - `ZMM1 = SPHERE A / STATE`
   - `ZMM2 = SPHERE B / TRANSACTION / DISTURBANCE`
4. **Облако событий** — это матрица `6x6 = 36` микро-состояний.
5. **Регистровая раскладка облака**:
   - `ZMM3 = A(+X) × B(all 6 axes)`
   - `ZMM4 = A(+Y) × B(all 6 axes)`
   - `ZMM5 = A(+Z) × B(all 6 axes)`
   - `ZMM6 = A(-X) × B(all 6 axes)`
   - `ZMM7 = A(-Y) × B(all 6 axes)`
   - `ZMM8 = A(-Z) × B(all 6 axes)`
6. **Фаза `1:3`** реализуется не дробью, а **трехтактным структурным циклом**.
7. **Зеркало / инверсия** проходит через `-Z` и маски зеркалирования.
8. **Fail-closed закон**: если состояние не самозамкнулось, оно не коммитится, а уходит в quarantine/zero.
9. **Запрет radius-0 commit**: система может касаться нуля как перехода, но не фиксировать в нём запись.

### 1.2. Что вторично и не должно ломать каркас

- 3x3 и 4x4 плоскости остаются как **подмодели кодирования**, но не как отдельные первичные ISA-объекты.
- 8-битные узлы важны как микрослой, но ядро пишется вокруг **512-битной сферы**.
- Пирамиды и кубы — не стартовая стадия, а расширения поверх уже работающих блоков `PHASE -> CLOUD -> FILTER -> ZERO-LOCK -> COMMIT`.

---

## 2. Как правильно составлять дорожную карту под 1000+ строк ASM

Чтобы roadmap не был “красивым текстом без выхода в код”, его нужно строить не по философским темам, а по **единицам сборки**.

### 2.1. Правило составления

Каждый блок дорожной карты обязан отвечать на 5 вопросов:

1. **Что хранится** — какие регистры, таблицы, буферы.
2. **Что делает** — какая топологическая операция происходит.
3. **Чем делает** — какие инструкции/макросы будут ядром реализации.
4. **Что выдает** — какой регистр/буфер считается результатом.
5. **Как проверяется** — какой инвариант обязан сойтись.

### 2.2. Формула блока

Для каждого крупного куска кода используем такой шаблон:

```text
БЛОК N
- Цель
- Входы
- Выходы
- Регистры
- Константы/таблицы
- Подпрограммы
- Инварианты
- Линейный бюджет строк
```

### 2.3. Главный принцип масштаба

Чтобы файл действительно дорос до `1000+` или `2000+` строк и при этом не развалился:

- не писать монолит “в одной простыне”;
- разбивать на блоки по 120–300 строк;
- для каждого блока иметь 3 слоя:
  - **данные/маски**,
  - **рабочие подпрограммы**,
  - **диагностика/инварианты**.

Именно это и наращивает код честно, а не искусственно.

---

## БЛОК VI. Движок фазы `1:3`
**Диапазон:** `1121–1460`

### Цель
Сделать фазу не дробью, а структурным трехтактным законом.

### Подблоки

#### VI.1. Прямая триада
- `(+X,+Y,+Z) => (+Y,+Z,+X)`
- три прохода
- сравнение с исходным состоянием

#### VI.2. Зеркальная триада
- `(-X,-Y,-Z) => (-Y,-Z,-X)`
- три прохода
- сравнение с исходным состоянием

#### VI.3. Фазовый verdict
- если после трех проходов состояние не вернулось — `reject`
- если вернулось — `carry = 0`

### Какие инструкции будут ядром
- скалярные `mov` для эталонного каркаса
- затем оптимизация через:
  - `vpermq`
  - `vpermt2q`
  - `vprolvq` / `vprorvq`

### Почему этот блок большой
Потому что тут нужно реализовать:
- каркасную версию,
- оптимизированную версию,
- сравнение,
- отказоустойчивость,
- режим диагностики.

## БЛОК XI. Фильтрация облака и редукция `36 -> 6 -> 1`
**Диапазон:** `2581–2920`

### Цель
Схлопнуть облако событий ступенчато, а не грубо.

### Этапы
1. `UNION` — всё, что вообще проявилось
2. `CORE` — устойчивое пересечение прямой и зеркальной ветви
3. `RESIDUE` — остаточная инъекция
4. `ROW REDUCE` — 6 строк до 6 кандидатов
5. `FINAL REDUCE` — 6 кандидатов до 1 результирующего вектора

### Почему так лучше
Если делать редукцию сразу `36 -> 1`, невозможно будет отлаживать, где именно сломалась геометрия. Ступенчатая редукция дает понятную диагностику.

## БЛОК XIII. ZERO-LOCK, COMMIT, QUARANTINE
**Диапазон:** `3281–3520`

### Цель
Окончательно решить: состояние живет, зануляется или уходит в карантин.

### Обязательная логика
- если `RESIDUE != 0` → reject
- если `CORE == 0` → reject
- если фазовая подпись невалидна → reject
- если всё сошлось → вписать `PHASE_ROOT` и `ECHO_ROOT`

### Подпрограммы
- `tlu_zero_lock`
- `tlu_commit`
- `tlu_quarantine`
- `tlu_commit_echo`

### Конечный инвариант
В память/результат попадает только состояние, которое:
- прошло фазу,
- прошло облако,
- прошло зеркало,
- не зависло в нуле.

- сравнение scalar vs vector path

### Важный порядок
Этот блок нельзя делать первым. Сначала correctness, потом density, потом speed.

---

## 4. Этапы сборки, а не только линии файла

Чтобы эта дорожная карта действительно привела к большому рабочему ASM, а не к бесконечному черновику, вот правильная последовательность сборки.

### Этап A — Канон и скелет (`~400–700 строк`)
- Пролог
- lane-map
- маски
- scratch-buffers
- `_start_tlu`
- `normalize`
- каркас `phase_engine`

### Этап B — Облако и fail-closed (`~700–1500 строк`)
- `expand_cloud`
- `self_closure_gate`
- `filter_cloud`
- `zero_lock`
- `commit/quarantine`

### Этап C — Сферическая арифметика (`~1500–2300 строк`)
- invert
- add
- multiply
- divide
- scoring

### Этап D — Диагностика и оптимизация (`~2300–4000+ строк`)
- self-tests
- macros
- optimized permutes
- debug/service sections

---

## 5. Что писать первым прямо в коде

Если переходить от roadmap к реальному `osiris_core.asm`, правильный порядок такой:

1. Зафиксировать `%define` и lane-map.
2. Собрать `section .data` с масками и таблицами.
3. Собрать `section .bss` со всеми scratch-буферами.
4. Написать `_start_tlu` как pipeline-скелет.
5. Реализовать `tlu_normalize_spheres`.
6. Реализовать **простую**, но корректную версию `tlu_phase_engine`.
7. Реализовать `tlu_expand_cloud` без оптимизации.
8. Реализовать `tlu_filter_cloud` и `tlu_zero_lock`.
9. Только потом переходить к `divide`, `resonate`, `vpermq`-ускорению.

Это важнее любой “красивой” арифметики.

---

## 6. Что не надо делать слишком рано

Чтобы проект не развалился, не стоит на первой итерации:

- сразу пытаться кодировать всю философию антивремени как отдельный ISA-слой;
- смешивать микрослой `8-bit` и макрослой `512-bit` в одном блоке;
- начинать с супер-оптимизированного `AVX-512`, пока не доказан корректный scalar/carrier path;
- строить гигантскую “универсальную” арифметику раньше, чем работает `phase -> cloud -> filter -> zero-lock`.

Ядро сначала должно **замыкаться**, а потом уже “быть красивым”.

---

## 7. Критерий готовности дорожной карты

Roadmap готов, если по нему можно без домыслов написать файл, в котором уже заранее понятны:

- диапазоны строк;
- состав `.data/.bss/.text`;
- названия ключевых меток;
- последовательность вызовов;
- инварианты на каждом этапе;
- тесты, которые доказывают корректность.

Эта дорожная карта этим требованиям уже соответствует.

---

## 8. Практический итог

Если делать `OSIRIS-ASM` правильно, то **ядро надо планировать не на 1000 строк, а сразу на диапазон `2600–4200+ строк`**.

Причина простая:
- меньше 1000 строк хватит только на красивый скелет;
- `1000–1800` — это каркас без настоящей диагностики;
- `2600+` — уже полноценное ядро с фазой, облаком, фильтром, zero-lock и сервисным слоем;
- `4000+` — зрелая версия с оптимизацией, тестами и таблицами.

Именно такой масштаб лучше всего соответствует материалам из ваших файлов.


# PRD — OSIRIS-ASM Roadmap Pass

## Original problem statement
Пользователь попросил составить дорожную карту для `OSIRIS-ASM`, чтобы она покрывала `1000+` строк ассемблерного кода и опиралась на материалы из файлов `32.txt` и `33 выжимка.txt`.

## Architecture decisions
- Канонический объект ядра: одна `512-bit sphere` в одном `ZMM` регистре.
- Канонический lane-map: `+X, +Y, +Z, -X, -Y, -Z, PHASE, ECHO`.
- Минимальная сцена исполнения: `ZMM0=C`, `ZMM1=A`, `ZMM2=B`.
- Облако событий: `6x6 = 36` микро-состояний в `ZMM3-ZMM8`.
- Центральный pipeline: `NORMALIZE -> PHASE(1:3) -> CLOUD -> FILTER -> ZERO-LOCK -> COMMIT`.
- Запрет radius-0 commit и fail-closed модель допущены как базовые инварианты.
